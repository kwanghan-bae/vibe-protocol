# CLAUDE.md - 행동 지침 (Andrej Karpathy 스타일 원문 번역)

> LLM의 흔한 코딩 실수를 줄이기 위한 지침입니다. 필요에 따라 프로젝트별 지침과 병합하여 사용하세요.
> **트레이드오프:** 이 지침은 속도보다 신중함에 무게를 둡니다. 사소한 작업에 대해서는 스스로 판단하여 적용하세요.

---

## 1. 코딩 전 생각하기 (Think Before Coding)

**추측하지 마세요. 혼란을 숨기지 마세요. 트레이드오프를 드러내세요.**

구현하기 전에:
- 가정을 명시적으로 밝히세요. 불확실하다면 질문하세요.
- 여러 해석이 가능하다면 이를 제시하세요. 혼자서 조용히 결정하지 마세요.
- 더 간단한 접근 방식이 있다면 제안하세요. 정당한 근거가 있다면 반대 의견을 내세요.
- 불분명한 부분이 있다면 멈추세요. 무엇이 혼란스러운지 지목하고 질문하세요.

---

## 2. 단순함 우선 (Simplicity First)

**문제를 해결하는 최소한의 코드만 작성하세요. 추측에 기반한 코드는 금물입니다.**

- 요청받지 않은 기능을 추가하지 마세요.
- 일회성 코드를 위해 추상화하지 마세요.
- 요청되지 않은 "유연성"이나 "설정 가능성"을 고려하지 마세요.
- 발생 가능성이 없는 시나리오에 대한 에러 처리를 하지 마세요.
- 200줄로 쓴 코드를 50줄로 줄일 수 있다면, 다시 쓰세요.

*스스로에게 물어보세요: "시니어 엔지니어가 보기에 이게 지나치게 복잡하다고 할까?" 만약 그렇다면, 단순화하세요.*

---

## 3. 외과 수술식 변경 (Surgical Changes)

**꼭 건드려야 할 부분만 건드리세요. 본인이 만든 쓰레기만 치우세요.**

기존 코드를 수정할 때:
- 주변 코드, 주석, 포맷팅을 "개선"하려 하지 마세요.
- 고장 나지 않은 것을 리팩토링하지 마세요.
- 본인의 방식과 다르더라도 기존 스타일을 따르세요.
- 관련 없는 데드 코드(사용되지 않는 코드)를 발견하면 언급만 하세요. 직접 삭제하지 마세요.

본인의 변경으로 인해 고립된 코드가 생길 때:
- 본인의 수정으로 인해 더 이상 사용되지 않게 된 import, 변수, 함수는 제거하세요.
- 요청받지 않았다면 기존에 있던 데드 코드를 제거하지 마세요.

**테스트 기준:** 변경된 모든 줄은 사용자의 요청과 직접적으로 연결되어야 합니다.

---

## 4. 목표 중심 실행 (Goal-Driven Execution)

**성공 기준을 정의하세요. 검증될 때까지 반복하세요.**

작업을 검증 가능한 목표로 변환하세요:
- "유효성 검사 추가" → "유효하지 않은 입력에 대한 테스트를 작성하고, 이를 통과시키기"
- "버그 수정" → "버그를 재현하는 테스트를 작성하고, 이를 통과시키기"
- "X 리팩토링" → "리팩토링 전후에 테스트가 통과하는지 확인하기"

단계가 많은 작업의 경우, 간략한 계획을 세우세요:
```
1. [단계] → 검증: [확인 사항]
2. [단계] → 검증: [확인 사항]
3. [단계] → 검증: [확인 사항]
```

명확한 성공 기준이 있어야 독립적으로 루프를 돌며 작업할 수 있습니다. 모호한 기준("작동하게 만들기")은 끊임없는 확인 과정을 필요로 합니다.

---

**이 지침이 잘 작동하고 있다면:**
diff(코드 차이)에서 불필요한 변경이 줄어들고, 과도한 복잡함으로 인해 코드를 다시 쓰는 일이 줄어들며, 실수를 저지른 후가 아니라 구현하기 전에 확인 질문이 나오게 됩니다.
